# React Application Build & Artifacts

This document explains how the React applications are built and what artifacts are produced for each S3 bucket.

## Build Overview

The project contains two independent React applications that share the same codebase structure but have different visual themes. Each app has its own Vite configuration, HTML entry point, and source directory.

| App | Source | Vite Config | HTML Entry | Build Output | Theme |
|-----|--------|-------------|------------|--------------|-------|
| Web1 (Blue) | `src-web1/` | `vite.config.web1.js` | `index-web1.html` | `web1/` | Purple gradient |
| Web2 (Green) | `src-web2/` | `vite.config.web2.js` | `index-web2.html` | `web2/` | Pink gradient |

## Build Commands

```bash
# Build the blue version
npm run build:web1    # runs: vite build --config vite.config.web1.js

# Build the green version
npm run build:web2    # runs: vite build --config vite.config.web2.js
```

## How the Build Works

Each Vite config does the following:

1. Takes the corresponding HTML entry file (e.g. `index-web1.html`) as the Rollup input.
2. Resolves the `@` alias to the app's source directory (e.g. `src-web1/`).
3. Bundles the React app (JSX, dependencies) into hashed JS and CSS chunks.
4. Outputs everything to the build directory (`web1/` or `web2/`).
5. A custom `rename-index` plugin runs after bundling — it renames the output HTML (e.g. `index-web1.html` → `index.html`). This is critical because API Gateway expects `index.html` in both buckets for blue/green switching to work.
6. `base: './'` ensures all asset references are relative paths, preventing cross-bucket asset loading when switching stages.

## Build Artifacts

After running `npm run build:web1`, the `web1/` directory contains:

```
web1/
├── index.html              # Entry HTML (renamed from index-web1.html)
└── assets/
    ├── index-[hash].js     # Bundled React app + dependencies
    └── index-[hash].css    # Extracted styles (if any)
```

The `web2/` directory has the same structure after `npm run build:web2`.

The `[hash]` in filenames is a content hash generated by Vite/Rollup for cache busting. It changes whenever the source code changes.

## What Goes Into Each S3 Bucket

The entire contents of each build directory are synced to the corresponding S3 bucket:

| S3 Bucket | Source Directory | Contents |
|-----------|-----------------|----------|
| `react-app-blue-{ACCOUNT_ID}` | `web1/` | `index.html`, `assets/index-[hash].js`, `assets/index-[hash].css` |
| `react-app-green-{ACCOUNT_ID}` | `web2/` | `index.html`, `assets/index-[hash].js`, `assets/index-[hash].css` |

The sync commands use `--delete` to remove any old files from the bucket, ensuring a clean deployment:

```bash
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)

# Sync blue artifacts
aws s3 sync web1/ s3://react-app-blue-${ACCOUNT_ID}/ --delete

# Sync green artifacts
aws s3 sync web2/ s3://react-app-green-${ACCOUNT_ID}/ --delete
```

## Relative Paths (`base: './'`)

The Vite config sets `base: './'` for both apps. This controls how asset URLs are written in the built `index.html`.

Without this setting, Vite defaults to `base: '/'`, which produces absolute paths:

```html
<!-- base: '/' (default) — BROKEN for this setup -->
<script type="module" src="/assets/index-abc123.js"></script>
```

With `base: './'`, paths become relative to the HTML file:

```html
<!-- base: './' — CORRECT -->
<script type="module" src="./assets/index-abc123.js"></script>
```

This is required as API Gateway serves the app under a stage prefix like `/prod/`, `/blue/`, or `/green/`. If the HTML references `/assets/...`, the browser requests `https://host/assets/index-abc123.js` — which doesn't exist at that path in API Gateway. With relative paths, the browser correctly requests `https://host/prod/assets/index-abc123.js`, which API Gateway proxies to the right S3 bucket.

## Hash-Based Routing (`HashRouter`)

Both React apps use `HashRouter` from `react-router-dom` instead of `BrowserRouter`. 

With `BrowserRouter`, navigating to `/prod/about` would send a request to API Gateway for a file called `about` in the S3 bucket — which doesn't exist. The server has no concept of React routes, so it returns a 404 or error.

With `HashRouter`, all routes live after the `#` in the URL:

```
https://{api-id}.execute-api.{region}.amazonaws.com/prod/#/
https://{api-id}.execute-api.{region}.amazonaws.com/prod/#/about
https://{api-id}.execute-api.{region}.amazonaws.com/prod/#/dashboard
```

The browser never sends the fragment (`#/about`) to the server. Every request hits `index.html`, and React handles the routing entirely on the client side. This means:

- No server-side routing or rewrite rules needed
- No catch-all proxy configuration for SPA fallback
- Deep links and page refreshes work out of the box
- The same `index.html` in each bucket handles all routes
